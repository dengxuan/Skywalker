# DI Auto-Registration Source Generator Contract

> Status: Draft for v2.0.0-preview.5  
> Tracking: #254, #278, #281
> Last updated: 2026-05-03

This contract defines the supported shape for the DI auto-registration source generator. The goal is to move `AddSkywalker()` from runtime feature-provider scanning toward compile-time generated service registration metadata while preserving the existing service registration semantics.

## Goals

- Keep the public `services.AddSkywalker()` entry point.
- Generate deterministic DI registration metadata for application services, domain services, repositories, and event handlers.
- Preserve existing `TryAdd` / `Replace` behavior, `ExposeServicesAttribute`, `SharedInstanceAttribute`, and keyed service semantics.
- Avoid runtime assembly/type scanning on the generated path.
- Keep explicit user registrations deterministic and easy to reason about.

## Current Runtime Model

`AddSkywalker()` currently:

1. Builds a `SkywalkerPartManager` from the entry assembly and Skywalker-related dependencies.
2. Scans parts for `IApplicationFeatureProvider<ServiceRegistrationFeature>` implementations.
3. Lets feature providers populate `ServiceRegistrationFeature.Services` and `ServiceRegistrationFeature.Replacements`.
4. Applies `TryAdd` for normal descriptors and `Replace` for replacement descriptors.
5. Calls `AddInterceptedServices()` so DynamicProxy can replace generated proxy candidates.

The generated model must behave the same for supported service shapes. The preview.5 implementation may keep the runtime feature-provider path as a non-AOT compatibility fallback, but generated metadata is the preferred path.

## User-Facing Attribute Model

The generator recognizes these attributes:

| Attribute | Target | Default lifetime | Purpose |
|---|---|---|---|
| `[Service]` | class | Scoped | General application service registration. |
| `[ApplicationService]` | class | Scoped | Application layer service registration. |
| `[Repository]` | class | Scoped | Custom repository registration. |
| `[EventHandler]` | class | Transient | Local/domain event handler registration. |

Each attribute should support:

- `ServiceLifetime Lifetime { get; init; }` when the category can override lifetime.
- `Type? ServiceType { get; init; }` for the common single-service override.
- A no-argument constructor for the default case.

Existing attributes remain valid and are honored by the generator:

- `ExposeServicesAttribute`
- `ReplaceServiceAttribute`
- `SharedInstanceAttribute`
- `KeyedServiceAttribute`

`ExposeServicesAttribute` remains the explicit multi-service override. If both the new attribute `ServiceType` and `ExposeServicesAttribute` are present, `ExposeServicesAttribute` wins because it already models multiple exposed services and `IncludeDefaults` / `IncludeSelf`.

## Convention Fallback

The generator supports attributes first and conventions second. Conventions can be disabled by MSBuild property:

```xml
<PropertyGroup>
  <SkywalkerConventionAutoRegister>false</SkywalkerConventionAutoRegister>
</PropertyGroup>
```

The implementation issue must expose this value to the generator with `CompilerVisibleProperty`.

Supported preview.5 conventions:

| Shape | Registration rule |
|---|---|
| Implements `IApplicationService` | Scoped application service. |
| Implements `IDomainService` | Scoped domain service. |
| Implements `IRepository` or a derived repository interface | Scoped repository. |
| Implements local/domain event handler interfaces | Transient event handler. |
| Class name ends with `AppService` or `ApplicationService` | Scoped application service when a matching service interface is found. |

Conventions should never register compiler-generated, abstract, static, open generic, or inaccessible implementation types.

## Service Type Resolution

For each implementation type, the generated registrar resolves exposed service types in this order:

1. `ExposeServicesAttribute.ServiceTypes`.
2. Attribute-level `ServiceType`.
3. Category-specific default interfaces discovered from the implemented interfaces.
4. Optional `IncludeDefaults` and `IncludeSelf` behavior from `ExposeServicesAttribute`.

Default interface discovery must exclude marker/framework interfaces such as:

- `IApplicationService`
- `IDomainService`
- `IRepository`
- `IInterceptable`
- `IDisposable`
- `IAsyncDisposable`

Repository discovery keeps concrete custom repository interfaces and may keep closed generic repository interfaces when that preserves existing custom repository override behavior.

## Generated Registrar Shape

Each consumer assembly that contains supported services gets one generated registrar:

```csharp
// <auto-generated/>
// This file was generated by Skywalker Source Generators.
// Do not modify this file manually.

#nullable enable
#pragma warning disable

[assembly: global::Skywalker.DependencyInjection.SkywalkerGeneratedDependencyInjectionRegistrationAttribute(
  typeof(global::Skywalker.Generated.__SkywalkerDependencyInjectionRegistrar),
  nameof(global::Skywalker.Generated.__SkywalkerDependencyInjectionRegistrar.AddSkywalkerGeneratedServices))]

namespace Skywalker.Generated;

internal static class __SkywalkerDependencyInjectionRegistrar
{
  public static Microsoft.Extensions.DependencyInjection.IServiceCollection AddSkywalkerGeneratedServices(
    Microsoft.Extensions.DependencyInjection.IServiceCollection services)
    {
        // generated TryAdd / Replace calls
    return services;
    }
}
```

Preview.5 uses an assembly metadata attribute, `SkywalkerGeneratedDependencyInjectionRegistrationAttribute`, that points `AddSkywalker()` to the generated registrar method. The method must be static, accept `IServiceCollection`, and return `IServiceCollection`. Invalid generated metadata fails with an actionable `InvalidOperationException` instead of being ignored.

## AddSkywalker Integration

`AddSkywalker()` should apply generated DI metadata before the runtime feature-provider fallback:

1. Discover generated registrars for the entry assembly and related Skywalker assemblies.
2. Invoke generated registration for assemblies that provide metadata.
3. Run the runtime feature-provider path as the compatibility fallback, including when no generated DI metadata exists.
4. Apply `AddInterceptedServices()` after service descriptors are finalized, matching current behavior.

Generated registrations must keep existing ordering rules:

- Normal descriptors use `TryAdd` semantics.
- `[ReplaceService]` descriptors use `Replace` semantics.
- User code may still register services manually before or after `AddSkywalker()` when it needs to override defaults.

## Supported preview.5 Service Shapes

- Non-abstract class implementations.
- Public or internal implementations when the generated registrar can legally reference them.
- Closed generic implementations where all type arguments are closed in the consuming assembly.
- One or more service interfaces discovered by convention or specified through `ExposeServicesAttribute`.
- Keyed services when the key value can be emitted as a compile-time constant or a supported literal.

## Known Limitations

- No open generic implementation registration in preview.5 unless #280 explicitly adds and tests it.
- No dynamic service type discovery that depends on runtime-only reflection.
- No registration for private nested implementation types.
- No source-generated factories for implementation factories supplied only at runtime.
- Keyed service keys that are not generator-emittable literals should report a diagnostic instead of being silently skipped.
- The DI generator does not require user service classes to be `partial`; it generates an external registrar and does not add members to user types.

## Diagnostics

DI auto-registration diagnostics use the `SKY1xxx` range defined in `source-generators-spec.md`.

Initial diagnostic candidates:

| ID | Condition | Severity |
|---|---|---|
| `SKY1001` | Implementation type is not accessible from generated code. | Warning |
| `SKY1002` | Service type cannot be assigned from implementation type. | Error |
| `SKY1003` | Open generic registration is not supported in preview.5. | Warning |
| `SKY1004` | Keyed service key cannot be emitted as generated code. | Warning |
| `SKY1005` | Multiple attributes define conflicting lifetimes or service types. | Warning |

Every diagnostic added in implementation issues must have a page under `docs/diagnostics/` before the preview.5 readiness issue closes.

## Test Expectations

Implementation PRs should cover:

- Attribute-driven registration for each supported category.
- Convention fallback enabled and disabled.
- `ExposeServicesAttribute` with `IncludeDefaults` and `IncludeSelf`.
- `SharedInstanceAttribute` across multiple service interfaces.
- `ReplaceServiceAttribute` ordering.
- Keyed service registration.
- Duplicate registration/idempotency behavior.
- Public, internal, nested, and unsupported private nested implementation types.
- Sample-level `AddSkywalker()` simplification.
